# Integration

Guide for consuming the inventory that `ansible-role-hetzner-cloud` produces after
it provisions infrastructure.

---

## Table of Contents

- [Dynamic inventory hand-off (recommended)](#dynamic-inventory-hand-off-recommended)
- [Static inventory fallback](#static-inventory-fallback)
- [Running downstream plays](#running-downstream-plays)

---

This role runs controller-side (on `localhost`, `connection: local`). After it
creates the requested servers, networks, firewalls and floating IPs, it hands off
to your downstream configuration plays via either the dynamic inventory plugin or
a static inventory file. Hosts are grouped by the `role` label applied at server
creation time, so your subsequent plays can target them by group.

### Inventory group contract

This role can emit hosts into arbitrary groups (via the dynamic plugin `groups:`
map or the static template) keyed off the labels you stamp on each server. The
group names below are illustrative examples — pick whatever names your downstream
plays consume:

| Group (example) | Label value |
|---|---|
| `server_nodes` | `role=control-plane` |
| `agent_nodes` | `role=worker` |
| `proxy_hosts` | `role=proxy` |
| `vault` | `role=vault` |

## Dynamic inventory hand-off (recommended)

After provisioning, use the `hetzner.hcloud.hcloud` inventory plugin to
automatically discover hosts by label. No static file to maintain; newly created
servers matching a label selector appear automatically.

Create a file named `hcloud.yml` (must end in `.hcloud.yml`) alongside your
playbooks:

```yaml
# hcloud.yml
plugin: hetzner.hcloud.hcloud
api_token: "{{ lookup('env', 'HCLOUD_TOKEN') }}"

# Filter to only servers on your private network (recommended for multi-tenant projects)
network: prod-net

# Group hosts by label so downstream plays can target them
groups:
  server_nodes: "labels.role == 'control-plane'"
  agent_nodes: "labels.role == 'worker'"
  proxy_hosts: "labels.role == 'proxy'"
  vault: "labels.role == 'vault'"

# Use private network IPs for SSH (requires network: above)
# hostvars_prefix defaults to 'hcloud_' since collection v5
compose:
  ansible_host: private_ipv4_address
```

Run with:
```bash
export HCLOUD_TOKEN="your-token"
ansible-inventory -i hcloud.yml --list
```

## Static inventory fallback

For air-gapped environments or CI pipelines that cannot use the dynamic plugin,
enable `hcloud_write_inventory: true` in your provisioning play. The role renders
`templates/hcloud_inventory.yml.j2` to `hcloud_inventory_path`:

```yaml
vars:
  hcloud_write_inventory: true
  hcloud_inventory_path: /tmp/hcloud_inventory.yml
```

The generated file groups servers by their `role` label, grouping hosts by label
as your downstream plays require:

```yaml
all:
  children:
    server_nodes:
      hosts:
        cp-01:
          ansible_host: 10.0.1.11
    agent_nodes:
      hosts:
        wk-01:
          ansible_host: 10.0.1.21
    proxy_hosts:
      hosts:
        px-01:
          ansible_host: 10.0.1.31
    vault:
      hosts:
        vault-01:
          ansible_host: 10.0.1.41
```

Note: the static inventory is generated from the `hcloud_provisioned_servers`
role fact set by `outputs.yml`. It only contains servers created in the same run
(not previously existing servers).

## Running downstream plays

Run your configuration plays against the produced groups with either hand-off.

With the dynamic inventory plugin (`hcloud.yml` is your hcloud inventory plugin
config — you provide this file; see
[hcloud inventory plugin docs](https://docs.ansible.com/ansible/latest/collections/hetzner/hcloud/hcloud_inventory.html)):

```bash
export HCLOUD_TOKEN="your-token"
ansible-playbook -i hcloud.yml your_playbook.yml
```

With the static inventory generated by the role (written to
`hcloud_inventory_path`), or by adapting
`examples/inventory/hcloud_static_example.yml` as a starting point:

```bash
ansible-playbook -i /tmp/hcloud_inventory.yml your_playbook.yml
```
